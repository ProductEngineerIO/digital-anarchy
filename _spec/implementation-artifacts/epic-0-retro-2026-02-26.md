# Epic 0 Retrospective — Fork Pattern Validation Spike

Facilitator: Bob (Scrum Master)

## Epic Summary

Epic 0 was a single-story spike to validate the fork architecture before committing to the full 10-epic implementation plan. The spike answered three pass/fail gates: (1) Does the Tier 2 hook work? (2) Do CSS variables propagate? (3) Does it deploy cleanly?

**Result: All 3 gates PASS.** The fork architecture is validated and ready for production use.

## What Went Well

1. **Fork architecture validated cleanly** — The Tier 2 dynamic `import()` hook proved sound. One line in `main.ts`, all logic in `src/fork/`. Graceful degradation confirmed — fork failure cannot crash the upstream app.

2. **CSS variable propagation exceeded expectations** — Required 10+ components across 3 categories; audit confirmed 20+ components across 3 categories respond to `--accent` override. Both dark and light themes covered.

3. **Surprise Tier 2 feasibility for panel reordering** — Expected `DEFAULT_PANELS` to require Tier 3 (upstream modification). Discovered the object is mutable (no `Object.freeze()`), so delete-reinsert or localStorage pre-seeding works at Tier 2. Significant de-risk for Epics 2 and 4.

4. **Zero regressions** — 144/144 existing tests pass. `tsc --noEmit` clean. Build succeeds.

5. **Test-first discipline held** — 9 unit tests covering config exports and DOM-mocking init behavior, including missing-document resilience.

6. **Strong story spec quality** — Thorough anti-pattern tables, architecture constraints, code specs, and clear acceptance criteria. Code review caught real issues that were fixed before merge.

## What Could Be Improved

1. **Gate 3 runtime verification delayed** — Build passed immediately but live Vercel Preview confirmation came a day later. In future, runtime verification should be part of the same work session.

2. **First-pass implementation gaps** — Code review caught Task 3.2 incompleteness (CSS probes for `--bg`, `--surface`) and a missing-document test gap. These should have been caught by the developer before submitting for review.

3. **Hardcoded-color components not yet categorized** — 12 components with hardcoded colors were documented but not classified as "intentional" vs "should migrate to tokens." Categorization added below for Epic 2 planning.

4. **No `test:fork` npm script yet** — Deferred to avoid second upstream file mod. Fork tests require `npx tsx --test src/fork/__tests__/*.test.mjs` until Story 1.2 adds the script.

## Key Decisions

| Decision | Rationale | Impact |
|---|---|---|
| Dynamic `import()` over static import | Ensures app fully initialized first; fork failure can't crash app | Foundation for all fork work |
| `<style id="fork-theme">` injection only | ARCH-22 — cascade control via document order | CSS branding pattern locked in |
| Main entry point only for spike | Scope control — secondary entry points deferred to Epic 2 | `settings.html`, `live-channels.html` need hooks later |
| Panel reordering is Tier 2 feasible | `DEFAULT_PANELS` is mutable, no `Object.freeze()` | De-risks Epic 2 and Epic 4 |
| `.test.mjs` with `tsx` runner | Matches existing test convention while supporting TS imports | Test pattern established for all fork stories |

## Appendix: Hardcoded-Color Component Categorization

Components that do NOT respond to CSS variable theming, categorized for Epic 2 planning:

### Intentional — Domain-Specific (Do NOT migrate)

These use hardcoded colors by design because the colors carry semantic meaning independent of brand:

| Component | Reason |
|---|---|
| PizzIntIndicator (DEFCON severity colors) | DEFCON levels have standardized color associations |
| InvestmentsPanel (status colors) | Green/red/yellow for gain/loss/neutral — universal financial convention |
| VerificationChecklist (verdict colors) | Pass/fail/pending — universal status convention |
| PIPELINE_COLORS config | Explicitly documented as "not theme-dependent" |
| DeckGLMap (`getOverlayColors()`) | ~30 RGBA tuples for WebGL layers — data visualization semantics, not branding |

### Should Migrate to Tokens (Epic 2 candidates)

These could reasonably adopt `--sm-*` tokens to align with fork branding:

| Component | Migration Approach | Complexity |
|---|---|---|
| MacroSignalsPanel (sparkline/gauge colors) | Replace with `--sm-chart-primary`, `--sm-chart-secondary` tokens | Low |
| CountryTimeline (lane colors) | Replace with `--sm-timeline-*` tokens | Low |
| ProgressChartsPanel (D3 stroke color) | Replace hardcoded stroke with `var(--sm-chart-stroke)` | Low |
| SignalModal (fallback `#ff9944`) | Replace fallback with `var(--sm-warning, #ff9944)` | Trivial |
| main.css misc (update-toast, beta-badge) | Replace inline hex with existing `var(--accent)` references | Trivial |

### Requires Investigation (Epic 2 spike)

| Component | Issue |
|---|---|
| Map.ts legend (inline style hex values) | Inline styles bypass CSS — may need JS intervention or attribute selectors |
| CountryBriefPage (SVG fills, print styles) | SVG `fill` attributes need `currentColor` pattern or CSS variable injection |
